# v1.2 优先：RBAC 权限 + DSL/Entity 锁粒度

> 用户优先级（1.1 终点之后）：  
> 1. **RBAC 权限控制**  
> 2. **用 DSL 或 Entity 定义控制锁的颗粒度**  
> 3. （口语「这个也需要」）与 1、2 并列的底座能力见文末 **§5 待对齐项**

原则仍成立：**业务 SQL 保持领域纯净；权限与锁由运行时按配置注入/施加。**

---

## 1. RBAC 权限控制（P0）

### 1.1 目标

- 至少两种角色：**老板 owner** / **店员 clerk**（可扩展）  
- **前端**：菜单、按钮、（可选）列展示  
- **后端**：**唯一可信**——page / action / query 调用拦截  
- **数据权限**：不写进每条 SQL 文本；用 **forcedParams / rowFilter**（见权限注入文档）

### 1.2 经典模型（最小表）

```text
lc_user
  user_id, login_name, password_hash, display_name, enabled, org_id?

lc_role
  role_code PK, name   -- owner, clerk

lc_user_role
  user_id, role_code

lc_permission
  perm_code PK          -- page:shop_sales | action:shop_post_sale_line | query:q_shop_products
  perm_type             -- page | action | query | field
  resource_code         -- shop_sales / shop_post_sale_line / ...
  description

lc_role_permission
  role_code, perm_code
```

**资源命名（现在就固定，避免返工）：**

| 类型 | 格式 | 例 |
|------|------|-----|
| 页面 | `page:{pageCode}` | `page:shop_products` |
| 动作 | `action:{actionCode}` | `action:shop_post_sale_line` |
| 查询 | `query:{queryCode}` | `query:q_shop_sales` |
| 字段 | `field:{entityCode}.{field}` | `field:entity_shop_product.cost_price` |

### 1.3 API 与会话

```text
POST /api/v1/auth/login     → token + profile
GET  /api/v1/auth/me        → user, roles, permissions[], dataScope, fieldDenies[]
Authorization: Bearer <token>
```

**后端拦截点（必须）：**

| 入口 | 检查 |
|------|------|
| `GET /pages/{pageCode}` | `page:{pageCode}` |
| `POST /queries/{code}/execute` | `query:{code}` |
| `POST /actions/{code}/execute` | `action:{code}` |
| `PUT` 配置类 API | 仅 owner / admin（配置态） |

**前端：**

- 侧栏：无 `page:*` 则不显示  
- 行/页按钮：无 `action:*` 则不渲染  
- 列：`fieldDenies` 隐藏成本价等（体验；后端列表仍可脱敏或拒绝字段）

### 1.4 角色默认矩阵（小店）

| 资源 | owner | clerk |
|------|--------|-------|
| 全部 shop 页面 | ✅ | ✅（可隐藏 SQL Repo / Factory） |
| SQL Repo / 配置态 | ✅ | ❌ |
| 商品 CRUD | ✅ | 查+改数量相关可收紧 |
| 成本价字段 | ✅ | ❌ |
| 确认入库/出库 | ✅ | ✅ 或仅出库 |
| 停用商品 | ✅ | ❌ |

### 1.5 数据权限（与 RBAC 同版最小做「范围类型」）

```json
"dataScope": {
  "type": "all" | "org" | "self",
  "orgId": 1
}
```

执行 Query/Action 时：

- `forcedParams` 写入 `org_id`（客户端不可覆盖）  
- **不要**改 SQL 仓库里每条语句手写角色  

表可先统一加可空 `org_id DEFAULT 1`，避免 2.0 大迁。

## 2. 锁粒度：DSL + Entity 定义（P0）

### 2.1 问题

仅写在某条过账 SQL 里的 `FOR UPDATE` 能用，但：

- 颗粒度不统一、难审计  
- 同一实体多动作容易锁顺序不一致 → 死锁  
- 配置作者不知道该锁什么  

### 2.2 两层定义（推荐）

#### A. Entity 级默认锁策略（实体语义）

`lc_entity_model` 扩展（或 `options_json` / `fields_json` 旁路）：

```json
{
  "entityCode": "entity_shop_product",
  "tableName": "shop_product",
  "primaryKey": "id",
  "lock": {
    "defaultMode": "row",
    "businessKeys": ["sku"],
    "advisoryPrefix": "product",
    "lockOrder": 100
  }
}
```

| 字段 | 含义 |
|------|------|
| `defaultMode` | `none` \| `row` \| `advisory` \| `row+advisory` |
| `businessKeys` | 业务键列（无 id 时用 sku 锁） |
| `advisoryPrefix` | `pg_advisory_xact_lock(hashtext(prefix \|\| key))` |
| `lockOrder` | 全局加锁顺序，防死锁（数字小先锁） |

#### B. Action / DSL 级覆盖（动作语义）

```json
{
  "type": "sqlTransaction",
  "actionCode": "shop_post_sale_line",
  "locks": [
    {
      "entity": "entity_shop_sale",
      "mode": "row",
      "keyFrom": { "id": { "from": "row", "field": "id" } }
    },
    {
      "entity": "entity_shop_product",
      "mode": "row+advisory",
      "keyFrom": { "sku": { "from": "row", "field": "sku" } }
    }
  ],
  "statements": [ ... ]
}
```

或简写引用实体默认：

```json
"locks": [
  { "entity": "entity_shop_sale", "useEntityDefault": true, "keyFrom": { "id": { "from": "row", "field": "id" } } },
  { "entity": "entity_shop_product", "useEntityDefault": true, "keyFrom": { "sku": { "from": "sql", "asset": "sql_resolve_sku_by_sale_id" } } }
]
```

`keyFrom` 与现有 action `bind` 同构，减少概念。

### 2.3 引擎执行顺序（ActionService）

```text
BEGIN
  1. 解析 locks[]（Action 覆盖 > Entity 默认）
  2. 按 lockOrder 排序
  3. 对每个锁：
       row      → SELECT pk FROM table WHERE ... FOR UPDATE
       advisory → SELECT pg_advisory_xact_lock(hashtext(...))
  4. 执行 statements[]（assert / write）
COMMIT / ROLLBACK
```

- **事务内锁**，无跨请求长锁（编辑锁另议，非最小店 P0）  
- SQL 仓库资产可继续保留 FOR UPDATE 作双保险，但**权威顺序以 DSL 为准**

### 2.4 小店默认颗粒度建议

| 动作 | 锁 |
|------|-----|
| 确认出库（行） | 锁 `shop_sale_line` 行 → 锁对应 `shop_product` 行（by sku） |
| 确认入库（行） | 锁 `shop_purchase_line` 行 → 锁 `shop_product` |
| 整单出库（1.2） | 锁该 `sale_no` 下所有草稿行 → 涉及 sku 排序后逐个锁 product |
| 停用商品 | 锁 product 行即可 |

### 2.5 与「只写在 SQL 里」的关系

| 阶段 | 做法 |
|------|------|
| 现在 / 过渡 | 过账 sqlAsset 内可先手写 `FOR UPDATE` |
| 1.2 目标 | **Entity.lock + Action.locks** 由引擎统一加锁 |
| 禁止 | 前端传锁；无事务的全局应用锁挡所有 API |

## 3. 实施切片（建议实现顺序）

### Slice A — RBAC 最小可跑

1. 表 + seed：owner/clerk + 权限矩阵  
2. 登录 / me / Bearer 过滤器  
3. 拦截 page / query / action  
4. 前端：根据 me 藏菜单与按钮  
5. demo 账号：`owner/owner`、`clerk/clerk`  

### Slice B — 锁 DSL 最小可跑

1. Entity `lock` JSON 字段（或 options_json）  
2. Action `locks[]` 解析  
3. ActionService 事务开头加锁  
4. 小店确认入/出库挂上 locks  
5. 并发测试：双请求确认同一行，仅一成功  

### Slice C — 加固（可并行）

- 整单过账 action  
- forcedParams `org_id`  
- 测库隔离、定时备份  

---

## 4. 非目标（本阶段仍不做）

- 完整 ABAC 规则引擎  
- Redis 分布式锁  
- 跨请求文档编辑锁 UI  
- 把 user/role 写进每条 SQL 资产

## 5. 「这个也需要」— 待你一句话对齐

你列了 1、2 后说「这个也需要」。请确认第 3 项是否指下面之一（可多选）：

| 代号 | 能力 |
|------|------|
| **C1** | 整单过账（同 sale_no / purchase_no 一次事务） |
| **C2** | 数据权限 forcedParams / org 范围（不只按钮 RBAC） |
| **C3** | 测试库与演示库隔离（防页面再丢） |
| **C4** | 定时备份 + assert 业务中文错误 |

**默认建议：** 1=RBAC、2=锁 DSL 为 P0；**C1+C3** 紧随，否则「有权限有锁」仍不够营业闭环与环境稳定。

---

## 6. 一句话

- **RBAC**：资源三元组 `page/action/query`，前端展示、后端强制。  
- **锁**：Entity 定默认颗粒度与顺序，Action DSL 声明本次锁谁；引擎在事务头统一施加。  
- **SQL 仓库**：继续不写角色；权限注入与锁编排都在运行时。  

确认第 3 点后，可直接按 Slice A → B 开工实现。
