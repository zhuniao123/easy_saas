# v1.2 优先：RBAC 权限 + DSL/Entity 锁粒度

> 用户优先级（1.1 终点之后，已对齐）：  
> 1. **RBAC 权限控制**（主线）  
> 2. **DSL / Entity 控制锁粒度**（主线）  
> 3. **C1 整单过账 + C2 数据权限 forcedParams/org**（主线，与 1、2 同版目标）  
> 支线：**C3 测库隔离、C4 定时备份 + 业务中文错误**  
> 工程原则：**优先成熟组件，少自研轮子**（见 §6）

原则仍成立：**业务 SQL 保持领域纯净；权限与锁由运行时按配置注入/施加。**

---

## 1. RBAC 权限控制（P0）

### 1.1 目标

- 至少两种角色：**老板 owner** / **店员 clerk**（可扩展）  
- **前端**：菜单、按钮、（可选）列展示  
- **后端**：**唯一可信**——page / action / query 调用拦截  
- **数据权限**：不写进每条 SQL 文本；用 **forcedParams / rowFilter**（见权限注入文档）

### 1.2 经典模型（最小表）

```text
lc_user
  user_id, login_name, password_hash, display_name, enabled, org_id?

lc_role
  role_code PK, name   -- owner, clerk

lc_user_role
  user_id, role_code

lc_permission
  perm_code PK          -- page:shop_sales | action:shop_post_sale_line | query:q_shop_products
  perm_type             -- page | action | query | field
  resource_code         -- shop_sales / shop_post_sale_line / ...
  description

lc_role_permission
  role_code, perm_code
```

**资源命名（现在就固定，避免返工）：**

| 类型 | 格式 | 例 |
|------|------|-----|
| 页面 | `page:{pageCode}` | `page:shop_products` |
| 动作 | `action:{actionCode}` | `action:shop_post_sale_line` |
| 查询 | `query:{queryCode}` | `query:q_shop_sales` |
| 字段 | `field:{entityCode}.{field}` | `field:entity_shop_product.cost_price` |

### 1.3 API 与会话

```text
POST /api/v1/auth/login     → token + profile
GET  /api/v1/auth/me        → user, roles, permissions[], dataScope, fieldDenies[]
Authorization: Bearer <token>
```

**后端拦截点（必须）：**

| 入口 | 检查 |
|------|------|
| `GET /pages/{pageCode}` | `page:{pageCode}` |
| `POST /queries/{code}/execute` | `query:{code}` |
| `POST /actions/{code}/execute` | `action:{code}` |
| `PUT` 配置类 API | 仅 owner / admin（配置态） |

**前端：**

- 侧栏：无 `page:*` 则不显示  
- 行/页按钮：无 `action:*` 则不渲染  
- 列：`fieldDenies` 隐藏成本价等（体验；后端列表仍可脱敏或拒绝字段）

### 1.4 角色默认矩阵（小店）

| 资源 | owner | clerk |
|------|--------|-------|
| 全部 shop 页面 | ✅ | ✅（可隐藏 SQL Repo / Factory） |
| SQL Repo / 配置态 | ✅ | ❌ |
| 商品 CRUD | ✅ | 查+改数量相关可收紧 |
| 成本价字段 | ✅ | ❌ |
| 确认入库/出库 | ✅ | ✅ 或仅出库 |
| 停用商品 | ✅ | ❌ |

### 1.5 数据权限（与 RBAC 同版最小做「范围类型」）

```json
"dataScope": {
  "type": "all" | "org" | "self",
  "orgId": 1
}
```

执行 Query/Action 时：

- `forcedParams` 写入 `org_id`（客户端不可覆盖）  
- **不要**改 SQL 仓库里每条语句手写角色  

表可先统一加可空 `org_id DEFAULT 1`，避免 2.0 大迁。

## 2. 锁粒度：DSL + Entity 定义（P0）

### 2.1 问题

仅写在某条过账 SQL 里的 `FOR UPDATE` 能用，但：

- 颗粒度不统一、难审计  
- 同一实体多动作容易锁顺序不一致 → 死锁  
- 配置作者不知道该锁什么  

### 2.2 两层定义（推荐）

#### A. Entity 级默认锁策略（实体语义）

`lc_entity_model` 扩展（或 `options_json` / `fields_json` 旁路）：

```json
{
  "entityCode": "entity_shop_product",
  "tableName": "shop_product",
  "primaryKey": "id",
  "lock": {
    "defaultMode": "row",
    "businessKeys": ["sku"],
    "advisoryPrefix": "product",
    "lockOrder": 100
  }
}
```

| 字段 | 含义 |
|------|------|
| `defaultMode` | `none` \| `row` \| `advisory` \| `row+advisory` |
| `businessKeys` | 业务键列（无 id 时用 sku 锁） |
| `advisoryPrefix` | `pg_advisory_xact_lock(hashtext(prefix \|\| key))` |
| `lockOrder` | 全局加锁顺序，防死锁（数字小先锁） |

#### B. Action / DSL 级覆盖（动作语义）

```json
{
  "type": "sqlTransaction",
  "actionCode": "shop_post_sale_line",
  "locks": [
    {
      "entity": "entity_shop_sale",
      "mode": "row",
      "keyFrom": { "id": { "from": "row", "field": "id" } }
    },
    {
      "entity": "entity_shop_product",
      "mode": "row+advisory",
      "keyFrom": { "sku": { "from": "row", "field": "sku" } }
    }
  ],
  "statements": [ ... ]
}
```

或简写引用实体默认：

```json
"locks": [
  { "entity": "entity_shop_sale", "useEntityDefault": true, "keyFrom": { "id": { "from": "row", "field": "id" } } },
  { "entity": "entity_shop_product", "useEntityDefault": true, "keyFrom": { "sku": { "from": "sql", "asset": "sql_resolve_sku_by_sale_id" } } }
]
```

`keyFrom` 与现有 action `bind` 同构，减少概念。

### 2.3 引擎执行顺序（ActionService）

```text
BEGIN
  1. 解析 locks[]（Action 覆盖 > Entity 默认）
  2. 按 lockOrder 排序
  3. 对每个锁：
       row      → SELECT pk FROM table WHERE ... FOR UPDATE
       advisory → SELECT pg_advisory_xact_lock(hashtext(...))
  4. 执行 statements[]（assert / write）
COMMIT / ROLLBACK
```

- **事务内锁**，无跨请求长锁（编辑锁另议，非最小店 P0）  
- SQL 仓库资产可继续保留 FOR UPDATE 作双保险，但**权威顺序以 DSL 为准**

### 2.4 小店默认颗粒度建议

| 动作 | 锁 |
|------|-----|
| 确认出库（行） | 锁 `shop_sale_line` 行 → 锁对应 `shop_product` 行（by sku） |
| 确认入库（行） | 锁 `shop_purchase_line` 行 → 锁 `shop_product` |
| 整单出库（1.2） | 锁该 `sale_no` 下所有草稿行 → 涉及 sku 排序后逐个锁 product |
| 停用商品 | 锁 product 行即可 |

### 2.5 与「只写在 SQL 里」的关系

| 阶段 | 做法 |
|------|------|
| 现在 / 过渡 | 过账 sqlAsset 内可先手写 `FOR UPDATE` |
| 1.2 目标 | **Entity.lock + Action.locks** 由引擎统一加锁 |
| 禁止 | 前端传锁；无事务的全局应用锁挡所有 API |

## 3. 主线范围（已确认）

| 代号 | 能力 | 轨道 |
|------|------|------|
| **1 RBAC** | 登录、角色、page/action/query/field、前后端分工 | **主线** |
| **2 锁 DSL/Entity** | Entity 默认颗粒度 + Action.locks + 事务头加锁 | **主线** |

| **C2** | 数据权限：forcedParams / org 范围（SQL 不写死角色） | **主线** |
| **C3** | 测试库与演示库隔离 | **支线** |
| **C4** | 定时备份 + assert/业务中文错误文案 | **支线** |

## 4. 实施切片（主线顺序）

### Slice A — RBAC（工具包优先，见 §6）

1. 引入安全组件 + 表/seed：owner、clerk  
2. login / me / Bearer（或 Token）过滤器  
3. 拦截 page / query / action  
4. 前端按 permissions 藏菜单与按钮  
5. demo：`owner` / `clerk`  

### Slice B — 锁 DSL/Entity

1. Entity `lock` 配置（options_json 或列）  
2. Action `locks[]`  
3. ActionService：BEGIN → locks → statements  
4. 小店行级确认入/出库挂 locks  
5. 并发单测  

### Slice C — C1 整单过账 + C2 数据范围

1. `shop_post_sale_order` / `shop_post_purchase_order`（按单号批量）  
2. 锁：单号下所有草稿行 + 涉及 sku 排序加锁  
3. 表预留/填充 `org_id`；Authz 注入 forcedParams  
4. clerk 仅本 org；owner 可 all  

### 支线（随时可插，不挡主线验收）

- **C3：** test profile 独立库 / Testcontainers；禁生产库整表 DELETE  
- **C4：** cron + `backup_db.sh`；action/assert 失败 message 字典化

## 6. 成熟工具包建议（减编码量）

当前栈：Spring Boot 3.2 + JDBC + PG + React。  
**能买就买、能配就不写**；自研只做「资源码 page/action/query」与「SQL 执行管线」的胶水。

### 6.1 安全 / RBAC（必选一类）

| 方案 | 优点 | 缺点 | 建议 |
|------|------|------|------|
| **Sa-Token** | 国内常用、登录/角色/权限注解/路由拦截代码量小、与 Boot3 集成快 | 国际资料少于 Spring Security | **1.2 首选（快）** |
| **Spring Security** | 官方、生态最大、过滤器链标准 | 定制资源权限样板代码偏多 | 若团队已熟可选用 |
| Keycloak / 外部 IdP | 企业级 SSO | 对最小店过重 | 后置 |

**推荐组合（Sa-Token 路径）：**

- `sa-token-spring-boot3-starter`：登录态、角色、权限字符串  
- 权限字符串直接用 `page:shop_sales`、`action:shop_post_sale_line`  
- 密码：`BCrypt`（Spring `spring-security-crypto` 即可，不必上全套 Security）  
- 前端：`/auth/me` 返回 `permissions[]`，本地 `can('action:xxx')`  

**若选 Spring Security：**

- `spring-boot-starter-security` + JWT（`spring-security-oauth2-jose` 或 jjwt）  
- 自写一个 `PermissionEvaluator` / Filter 查 `lc_role_permission`  
- 代码量通常 > Sa-Token，但「零第三方权限框架」时更干净  

### 6.2 锁（少引库）

| 需求 | 建议 |
|------|------|
| 过账一致性 | **PostgreSQL 原生**：`FOR UPDATE` + `pg_advisory_xact_lock`（引擎生成，不必 Redisson） |
| 定时备份防重入 | **ShedLock** 或 **Spring Integration** 轻量；单机 cron 也可文件锁 |
| 分布式多实例锁 | 以后再上 Redis/Redisson，1.2 不做 |

自研只做：**解析 Entity/Action 的 locks[] → 调 PG**。

### 6.3 数据权限 C2（少引库）

- 不引「数据权限中台」  
- 自研薄层：`AuthContext` + `forcedParams` 合并进 NamedParameterJdbcTemplate 调用前  
- 可选后期：PG **RLS**（仍是数据库能力，不是 Java 包）  

### 6.4 整单过账 C1（配置为主）

- 基本不引新库：新 `lc_action` + 多条/循环 sqlAsset + locks  
- 若要「语句模板循环」：引擎支持 `statements[].forEach: { from: "query", queryCode: "q_draft_lines" }` 可后加；首版可用 **一条 SQL 按单号批量 UPDATE/INSERT** 完成整单  

### 6.5 其它值得引入的「少写代码」

| 用途 | 组件 |
|------|------|
| 表结构演进 | **Flyway** 或 **Liquibase**（替代长期堆 schema.sql 手工） |
| API 文档 | **springdoc-openapi** |
| 参数校验 | **spring-boot-starter-validation**（`@Valid`） |
| 映射 DTO | 不必 MapStruct（继续 Map/jsonb 即可） |
| 前端请求鉴权 | axios/fetch 拦截器挂 Token（自写 20 行） |
| 前端路由守卫 | 若上 react-router 再加；现 Tab 壳按 permissions 过滤即可 |

### 6.6 明确不建议 1.2 引入

| 组件 | 原因 |
|------|------|
| Flowable/Camunda | 过重，不是最小店 |
| 完整 Shiro 双栈 | 与 Sa-Token/Security 重复 |
| Redisson | 单库未必要 |
| 自研完整安全框架 | 重复造轮子 |

### 6.7 依赖落位示意（Sa-Token 路径）

```xml
<!-- pom 示例，实现时再钉版本 -->
<dependency>
  <groupId>cn.dev33</groupId>
  <artifactId>sa-token-spring-boot3-starter</artifactId>
</dependency>
<dependency>
  <groupId>org.springframework.security</groupId>
  <artifactId>spring-security-crypto</artifactId>
</dependency>
```

锁与 forcedParams：**零新中间件**，PG + 现有 ActionService/QueryEngine 扩展。

---

## 7. 一句话

- **主线：** RBAC + 锁 DSL/Entity + **整单过账 C1** + **数据范围 C2**。  
- **支线：** 测库隔离 C3、定时备份与中文错误 C4。  
- **减负：** 权限用 **Sa-Token（或 Spring Security）**；锁用 **PG 原生**；业务仍 **DSL + SQL 仓库**，不写领域过账 Java。  

下一步实现默认顺序：**Slice A（Sa-Token RBAC）→ B（锁）→ C（C1+C2）**；支线可随时插。
