## 3. 数据库可替代性评估（现阶段）

### 3.1 结论（先说清楚）

| 问题 | 答案 |
|------|------|
| **能否换库？** | **能换，但不是零成本。** 业务 SQL 与元数据层都「轻度依赖」PostgreSQL 习惯用法。 |
| **是否用了 PG 高级特性？** | **几乎没有**（无分区表、无物化视图、无 LISTEN/NOTIFY、无 advisory 锁实现、无 RLS）。 |
| **卡点是什么？** | 元数据 **JSONB**、**ILIKE**、**ON CONFLICT**、**BIGSERIAL/TIMESTAMPTZ**、部分 **布尔字面量**、方言 SQL 生成。 |
| **最现实迁移目标** | **MySQL 8+ / MariaDB**（需改方言与 JSON）；**SQL Server** 更贵；**H2** 仅测试。 |
| **平台库 vs 业务库** | 1.6 后：**平台元数据库可继续 PG**；**业务库**通过 `data_source_code` 接其它 JDBC——这是更优路径，不必强行「全栈换库」。 |

### 3.2 已使用的「方言敏感」点

| 用法 | 出现位置 | PG 特有程度 | 移植策略 |
|------|----------|-------------|----------|
| `JSONB` + `::jsonb` | `lc_*` 元数据、日志 details | 高（MySQL 用 JSON） | Dialect 映射类型；或 TEXT+应用层 JSON |
| `ILIKE` | QueryEngine 文本 filter | 中（MySQL `LIKE` + lower） | Dialect.likeIgnoreCase() |
| `ON CONFLICT … DO NOTHING` | RBAC seed、权限 upsert | 中（MySQL `INSERT IGNORE` / `ON DUPLICATE`） | 统一 Upsert API |
| `BIGSERIAL` / `SERIAL` | 主键 | 中（AUTO_INCREMENT / IDENTITY） | schema 方言文件 |
| `TIMESTAMPTZ` / `now()` | 时间列 | 低～中 | TIMESTAMP + UTC 约定 |
| `BOOLEAN` | enabled 等 | 低（MySQL TINYINT） | 映射 |
| `GENERATED … STORED` | shop demo amount | 中 | 应用层计算或触发器 |
| `INTERVAL` / demo seed | demos | 低 | 演示 SQL 可按库拆分 |
| `FOR UPDATE` / advisory | **规划中** 锁 DSL | 中～高 | 锁 SPI 按方言实现 |
| 标识符 `"table"` 双引号 | factory 建表 | 中（MySQL 反引号） | Dialect.quote() |

**未使用（说明「高级特性负担低」）：**

- 物化视图、CTE 递归生产路径、分区表、逻辑复制  
- `pg_advisory_xact_lock`（仅文档规划）  
- RLS 行安全策略  
- LISTEN/NOTIFY  

### 3.3 分层结论

```text
┌─────────────────────────────────────────┐
│  业务 SQL 资产（用户配置）                 │  ← 用户自己负责方言；换库需改 SQL
├─────────────────────────────────────────┤
│  引擎生成 SQL（filter / CRUD / 锁）        │  ← 平台必须 Dialect
├─────────────────────────────────────────┤
│  元数据 schema（lc_*）                     │  ← 平台维护多方言 DDL 或抽象类型
├─────────────────────────────────────────┤
│  JDBC 驱动 + 连接池                        │  ← 1.6 多数据源已预埋
└─────────────────────────────────────────┘
```

**推荐产品策略（2.0）：**

1. **平台元数据库**：继续默认 PostgreSQL（JSONB 舒服）；可选 MySQL 方言包。  
2. **业务库**：通过数据源目录挂载 **任意 JDBC**；不承诺跨方言自动改写用户 SQL。  
3. **引擎生成 SQL**：只保证 **Dialect 覆盖的 CRUD/filter/锁** 可移植。  
4. **不要**宣称「一键从 PG 迁到 Oracle 零改 SQL」——不诚实。

### 3.4 可替代性评级

| 目标库 | 作为平台元数据库 | 作为业务库（用户 SQL） | 工作量 |
|--------|------------------|----------------------|--------|
| PostgreSQL | ⭐ 原生 | ⭐ 原生 | 0 |
| MySQL 8 | 🟡 可行（JSON/upsert/ILIKE） | 🟡 用户 SQL 自理 | 中（2～4 人周 Dialect+DDL） |
| MariaDB | 同 MySQL | 同 MySQL | 同左 |
| SQL Server | 🟡 可行 | 🟡 | 中高 |
| Oracle | 🟠 贵 | 🟠 | 高 |
| SQLite/H2 | 🔴 仅单测/演示 | 🔴 | 仅冒烟 |

## 4. 后台日志体系规划

### 4.1 现状（已有碎片）

| 表/组件 | 用途 | 缺口 |
|---------|------|------|
| `lc_error_log` + ErrorLogService | API 异常落库 | 无 requestId、无 userId、无 ds_code、管理 UI 简陋 |
| `lc_query_log` | 查询成功/失败与耗时 | 无 params 摘要、无慢查询看板 |
| `lc_action_log` | 动作执行 | 需统一字段与检索 |
| `lc_client_log` | 前端埋点 | 与后端审计未打通 |
| GlobalExceptionHandler | 写 error_log | 缺关联 trace |
| 应用日志文件 | Spring 默认 | 未结构化、无租户/用户 MDC |

### 4.2 目标模型（2.0）

**三类日志，勿混为一谈：**

| 类型 | 谁写 | 用途 | 保留 |
|------|------|------|------|
| **A. 运行审计 Audit** | 后端拦截器/服务 | 谁在何时对何资源做了何操作 | 90d+ |
| **B. 执行遥测 Telemetry** | Query/Action 引擎 | 耗时、成功、ds_code、行数 | 14～30d |
| **C. 诊断 Diagnostic** | 异常/堆栈 | 排障 | 30d |

建议统一信封字段：

```text
log_id, ts, level,
request_id, user_id, login_name, org_id,
source (api|query|action|client|system),
ds_code,
resource_type, resource_code,
success, duration_ms,
message, detail_json
```

### 4.3 实施步骤

1. **E1** 引入 `RequestIdFilter` + MDC（userId 登录后注入）  
2. **E2** 扩展 `lc_error_log` / `lc_query_log` 字段：request_id, user_id, ds_code  
3. **E3** 管理台「运维日志」页：筛错误、慢查询（>N ms）、按用户  
4. **E4** Logback JSON 控制台/文件（与 DB 审计互补；DB 可查询，文件可接 ELK）  
5. **E5** 高基数 detail 截断，防日志库爆炸  
6. **E6** 可选：异步写日志（队列），避免拖慢主路径  

### 4.4 不做

- 把业务结果集整包打进日志  
- 用日志代替 RBAC 审计的唯一来源  
- 未登录就记敏感 SQL 参数明文（需脱敏）

## 5. 建议实施顺序（合并甘特）

```text
现在(1.5收官)
    │
    ├─1.6a── A1/A2 多库运行时路由 + E5 ds_code 进查询日志
    ├─1.6b── B3 C2 数据范围 + B1 锁 DSL（业务正确性）
    ├─1.6c── E1/E2/E3 日志产品化
    │
    ├─2.0a── C1 masterDetail + C2 openPage
    ├─2.0b── D1/D2 缓存与 Tab 性能
    ├─2.0c── F1 Dialect（若要正式支持 MySQL 元数据库）
    └─2.0d── D3/D4 脚本与插件
```

**原则：** 先 **正确（权限/锁/多库路由）**，再 **快（缓存）**，再 **大（主从 UI）**，最后 **宽（多方言元数据库）**。

## 6. 风险与依赖

| 风险 | 缓解 |
|------|------|
| 多库路由半接入导致写错库 | 默认 default；非 default 强制显式；试连+只读标记 |
| 用户 SQL 绑 MySQL 却写 PG 语法 | 文档声明「业务 SQL 方言自负」；模板按 ds 标注 dialect |
| 日志表暴涨 | TTL 任务、采样、detail 截断 |
| 锁 DSL 先写死 PG | 锁 SPI 接口，PG 实现优先 |
| 测试污染演示库 | B4 测库隔离 |
